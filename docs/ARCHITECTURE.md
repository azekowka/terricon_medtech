# Архитектура MedServicePrice.kz

## Слои данных (TЗ 3.1)

Данные проходят три изолированных слоя, что обеспечивает аудит и воспроизводимость:

1. **Источники** — парсеры (`app/parsers/`). Каждый источник реализует `collect()` и
   возвращает список `RawRecord`. Поддерживаются форматы **HTML** (`live/` + `html_extract.py`),
   **PDF / XLSX / DOCX** (`file_extract.py` + `fixtures.py`) и **реальные прайс-листы клиник
   PDF/DOCX/XLSX/XLS** (`real_extract.py` + `real_prices.py`, колоночное + построчное
   распознавание) — все проходят единый пайплайн. Парсеры вежливы (задержки, Crawl-delay из
   robots.txt) и отказоустойчивы: исключение в одном источнике логируется и не останавливает
   остальные.

2. **Raw-слой** (`raw_prices`) — сырые строки «как спарсили», до нормализации. Хранятся
   ≥90 дней для аудита (TЗ 4). Никогда не редактируются.

3. **Нормализованный слой** (`prices`) — очищенные, дедуплицированные записи, к которым
   обращается UI. Привязаны к канонической услуге из справочника.

```
RawRecord ──▶ raw_prices ──▶ [normalizer] ──▶ prices ──▶ price_history
                                   │
                                   └─(не сопоставлено)─▶ unmatched_queue
```

## Нормализация (TЗ 3.2)

Справочник услуг — **официальный реестр из 1230 услуг (122 специальности)** с тарифными
кодами (`data/source/services_reference.xlsx`), импортируемый `scripts/build_dictionary.py`:
категория выводится из префикса тарифного кода (A=приём, B=лаборатория, C=диагностика,
D=процедура), а наши кураторские синонимы/цены накладываются на совпадающие позиции
(rapidfuzz). `app/normalization/normalizer.py` строит индекс из таблицы `services`: каждое
каноническое имя и каждый синоним нормализуются (`normalize_text`: нижний регистр, ё→е,
удаление пунктуации и «шумовых» слов). Сопоставление:

1. **Точное** совпадение по индексу → score 100.
2. **Нечёткое** (`rapidfuzz.WRatio`) ≥ порога (`MATCH_THRESHOLD`, по умолч. 86) → привязка.
3. Ниже порога → запись в `unmatched_queue` с лучшей подсказкой и счётчиком повторов.

Администратор в `/admin` привязывает элемент очереди к услуге: синоним «дообучается»
(добавляется в справочник), а существующие цены **переустанавливают** связь — петля
обратной связи замыкается.

## Дедупликация (TЗ 3.1)

Ключ `dedup_key = sha1(clinic_id | raw_name | source)`. При повторном парсинге запись
обновляется (upsert), а не дублируется. Если цена изменилась — в `price_history`
добавляется новая точка, `parsed_at` обновляется.

## Свежесть данных (TЗ 4)

Каждая запись хранит `parsed_at`. Данные старше `STALE_DAYS` (30) не считаются актуальными:
по умолчанию скрыты в поиске (чекбокс «показывать устаревшие»), помечены в карточках.
`mark_stale_inactive()` снимает флаг `is_active`. Планировщик (`scheduler.py`) ежедневно
перепарсивает, помечает устаревшее и чистит raw-слой старше retention.

## Поиск (TЗ 3.3, отклик <3 c)

`app/search.py`:
* **Автодополнение** — матч запроса по справочнику (имя + синонимы), ранжирование по
  совпадению и числу предложений.
* **Поиск предложений** — резолв запроса в каноническую услугу (через нормализатор),
  затем фильтры (город, категория, цена, рейтинг, онлайн-запись), сортировки (цена,
  дата, рейтинг, расстояние через гаверсинус), статистика min/avg/max.

Объём данных (сотни строк) даёт отклик ~10–50 мс; индексы на `service_id`, `category`,
`is_active`, `city`.

## Масштабируемость (TЗ 4)

Добавление источника = новый класс-парсер + строка в `registry.py`. Ядро (нормализация,
дедуп, история, API) не меняется. СУБД переключается одной переменной `DATABASE_URL`
(SQLite ↔ PostgreSQL) без изменения кода.
