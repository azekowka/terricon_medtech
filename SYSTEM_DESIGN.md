# System Design — MedServicePrice.kz

> Технический документ: как устроено решение (бэкенд, фронтенд, парсер, нормализация,
> база данных, Docker) **в мельчайших деталях**, и почему по каждому пункту
> Технического задания (ТЗ, Кейс 1) выбранные решения технически оптимальны.

**MedServicePrice.kz** — агрегатор и сравнение цен на медицинские услуги в Казахстане
(«Aviasales для медицины»): один поиск вместо обхода десятков сайтов клиник. Система
**собирает (парсит) прайсы → нормализует их к единому собственному справочнику услуг →
отдаёт через API → показывает в адаптивном веб-интерфейсе** с поиском, фильтрами,
сравнением, картой, историей цен, подписками, а также каталогом врачей и заболеваний.

## 1. Архитектура и поток запроса

### 1.1 Слои системы

Решение — это классический трёхзвенный сервис с чётким разделением **сбор → хранение →
выдача → представление**. Ключевой архитектурный приём — **двухслойное хранение данных**
(сырой слой отдельно от нормализованного) и **единый контракт `RawRecord`**, через который
проходит любой источник, поэтому добавление нового источника не затрагивает ядро.

```
┌──────────────────────────────────────────────────────────────────────────┐
│  ИСТОЧНИКИ (плагины-парсеры, общий контракт RawRecord)                       │
│  • seed      — детерминированный генератор из СВОЕГО справочника услуг        │
│  • fixtures  — образцовые публичные прайсы клиник (PDF / XLSX / DOCX)         │
│  • live      — веб-скрейперы (kdl / invitro / helix), HTML, по --live         │
│  • real      — подключаемый адаптер прайс-файлов клиник (PDF/DOCX/XLS/XLSX)   │
└───────────────┬──────────────────────────────────────────────────────────┘
                │ list[RawRecord]
                ▼
        ┌───────────────┐   сырой слой   ┌──────────────┐  нормализация  ┌──────────────┐
        │  pipeline.py   │ ─────────────▶ │  raw_prices   │ ─────────────▶ │   prices      │
        │ run_source()   │  (аудит ≥90д)  │  (RawPrice)   │ (справочник)   │ (нормализ.,   │
        └──────┬────────┘                 └──────────────┘                 │  дедуп, UI)   │
               │ нет совпадения                                            └──────┬───────┘
               ▼                                                                  │ история цен
        unmatched_queue ◀── suggest()                                     price_history
        (ручная разметка, TZ 3.2)                                         (TZ 3.4)
               │
               │  ParseLog (журнал ошибок: source + причина + счётчики, TZ 3.1)
               ▼
        ┌──────────────────────────────────────────────────────────────────┐
        │  FastAPI (11 роутеров /api/*)  — поиск, фильтры, сравнение, карта,  │
        │  история, подписки, врачи, болезни, ассистент, админка             │
        └───────────────┬──────────────────────────────────────────────────┘
                        │ JSON (fetch, cache: no-store)
                        ▼
        ┌──────────────────────────────────────────────────────────────────┐
        │  Next.js 14 (App Router, TS, Tailwind) — адаптивный веб-интерфейс   │
        │  MapLibre GL (карта), Recharts (история), i18n RU/KK/EN            │
        └──────────────────────────────────────────────────────────────────┘
```

### 1.2 Жизненный цикл типичного запроса «найти цену на услугу»

1. Пользователь печатает в `SearchBar`; через **debounce 180 мс** фронт зовёт
   `GET /api/search/autocomplete?q=…` → бэкенд скорит услуги по имени+синонимам и
   возвращает подсказки, отсортированные по релевантности и числу предложений.
2. Пользователь выбирает услугу → переход на `/search?service_id=…`. Фронт (**debounce
   250 мс**, отмена устаревших ответов) зовёт `GET /api/search?service_id=…&city=…&sort=…`.
3. Бэкенд (`search_offers`) джойнит `Price`↔`Clinic`, фильтрует по `is_active` + услуге +
   город/категория/цена/рейтинг/онлайн, **отбрасывает данные старше 30 дней** (ТЗ 4),
   считает расстояние (haversine, если есть геолокация), сортирует, считает
   `min/avg/max`, пагинирует и отдаёт `{service, stats, offers}`.
4. Фронт рисует список `OfferCard` (цена, клиника, адрес, часы, бейдж свежести, ссылка на
   источник), отмечает «дешевле всех», позволяет выбрать клиники → `/compare`.
5. На `/compare` бэкенд переиспользует тот же `search_offers`, фронт группирует филиалы
   сети в раскрываемые строки-аккордеоны и рисует график истории цен (Recharts).

### 1.3 Технологический стек и почему именно он

| Слой | Технология | Обоснование |
|---|---|---|
| Парсинг | **Python** (pdfplumber, python-docx, openpyxl, xlrd, BeautifulSoup+lxml, httpx, rapidfuzz) | ТЗ 5 рекомендует Python; в Python лучшая экосистема извлечения из PDF/DOCX/XLS и нечёткого сопоставления (rapidfuzz — на C, очень быстрый). |
| Бэкенд / API | **FastAPI 0.115** + **SQLAlchemy 2.0** + **Pydantic v2** | ТЗ 5 прямо называет FastAPI: async, авто-`/docs` (OpenAPI), строгая типизация запросов/ответов, минимум бойлерплейта. SQLAlchemy 2.0 даёт **кросс-СУБД** один код на SQLite и PostgreSQL. |
| База данных | **SQLite** (dev, zero-infra) / **PostgreSQL 16** (prod, docker) | ТЗ 5 рекомендует PostgreSQL. Кросс-СУБД движок: «из коробки» работает без инфраструктуры (SQLite), а в докере поднимается боевой Postgres — одной переменной `DATABASE_URL`. |
| Расписание | **APScheduler** | ТЗ 5 называет его; in-process cron без отдельного брокера (Celery/Redis избыточны для ежедневного респарса MVP). |
| Фронтенд | **Next.js 14 (App Router)** + **TypeScript** + **Tailwind 3.4** + **pnpm** | ТЗ 5 называет Next.js с pnpm. App Router: серверные компоненты для SEO/скорости + клиентские для интерактива; Tailwind — быстрый адаптивный UI (ТЗ 3.3). |
| Карта | **MapLibre GL** (бесплатный векторный стиль CARTO) | ТЗ 3.4 допускает Leaflet/Google Maps; MapLibre даёт «авиасейлс»-вид (векторные тайлы) **без API-ключа** и без затрат. |
| Графики | **Recharts** | Декларативные React-графики для истории цен (ТЗ 3.4). |
| Деплой | **Docker + docker-compose** | ТЗ 5: один `docker compose up` поднимает Postgres + бэкенд + фронт. |

## 2. Бэкенд: ядро, конфигурация, БД и схема данных (ТЗ 2.2)

Версия пакета — `__version__ = "1.0.0"` (`backend/app/__init__.py`). Весь бэкенд — это
тонкие роутеры поверх SQLAlchemy-моделей; вся «тяжёлая» логика (ранжирование, нормализация,
парсинг) вынесена в `search.py` / `normalization/` / `parsers/` и переиспользуется.

### 2.1 Точка входа и жизненный цикл (`main.py`)

`FastAPI(title="MedServicePrice.kz API", version="1.0.0", lifespan=lifespan)`.
Асинхронный `lifespan` при старте по порядку:

1. `init_db()` — создаёт все таблицы из метаданных (без Alembic — идемпотентный `create_all`).
2. `bootstrap_if_empty(db)` — наполняет пустую БД (см. 2.6); результат логируется.
3. `start_scheduler()` — запускает опциональный APScheduler-cron.
4. `yield` (приложение обслуживает запросы); на остановке — `shutdown_scheduler()`.

Подключены **11 роутеров** одним циклом `app.include_router(...)`: `meta, search, services,
clinics, history, admin, subscriptions, doctors, map, assistant, illnesses`. CORS —
`CORSMiddleware` с `allow_origins=settings.cors_list`. Служебные эндпоинты:
`GET /api/health → {status, version}` и `GET /` (баннер со ссылкой на `/docs`).

### 2.2 Конфигурация (`config.py`) — `pydantic-settings`

Любое поле переопределяется переменной окружения или `.env` (`extra="ignore"`). Singleton
`settings = Settings()` импортируется повсюду.

| Параметр | Значение по умолчанию | Назначение / ТЗ |
|---|---|---|
| `database_url` | `sqlite:///./medservice.db` | DSN; docker задаёт Postgres |
| `cors_origins` | `*` | разрешённые источники |
| `data_dir` | `<repo>/data` | где лежат JSON-данные |
| `usd_kzt_rate` | `470.0` | конвертация USD→KZT (ТЗ 2.2) |
| `stale_days` | `30` | данные старше — не «актуальны» (ТЗ 4) |
| `raw_retention_days` | `90` | хранение сырого слоя для аудита (ТЗ 4) |
| `match_threshold` | `86` | порог нечёткого матчинга; ниже → очередь разметки (ТЗ 3.2) |
| `enable_scheduler` | `False` | включить ежедневный cron |
| `scheduler_cron_hour` | `3` | час cron (03:00 UTC) |
| `openai_api_key/model/base_url` | `""` / `gpt-4o-mini` / `api.openai.com/v1` | ассистент; пусто → детерминированный движок |

### 2.3 Движок БД (`db.py`) — кросс-СУБД

```python
_is_sqlite = settings.database_url.startswith("sqlite")
engine = create_engine(settings.database_url,
    connect_args={"check_same_thread": False} if _is_sqlite else {},
    pool_pre_ping=True, future=True)
```

- Для **SQLite** добавляется `check_same_thread=False` (БД-файл шарится между потоками
  FastAPI и APScheduler).
- Для **PostgreSQL** — обычный пул.
- `pool_pre_ping=True` проверяет соединение перед использованием (отказоустойчивость, ТЗ 4);
  `future=True` — API SQLAlchemy 2.0.
- `SessionLocal` (explicit commit, без autoflush), `Base(DeclarativeBase)`, `get_db()` —
  request-scoped зависимость FastAPI.

### 2.4 Схема данных — реализация ТЗ 2.2 и двухслойная модель (ТЗ 3.1)

**Ключевая идея:** сырьё падает сначала в `raw_prices` (неизменяемый слой для аудита,
хранится ≥90 дней), затем нормализатор пишет дедуплицированную, привязанную к услуге строку
в `prices` — единственную таблицу, которую читает поиск/UI. Это разделяет «что мы спарсили»
и «что показываем», обеспечивает идемпотентный upsert по `dedup_key` и независимые сроки
жизни слоёв.

**`Service`** — единый справочник услуг (ТЗ 3.2): `id`(uuid, PK), `code`(unique),
`name`(idx), `category`(idx), `specialty`(idx), `tarif_code`, **`synonyms`(JSON)**,
`base_price_kzt`, `duration_days`, `created_at`. Покрывает требование ТЗ 3.2
*id / name / synonyms / category*.

**`Clinic`** — клиника/филиал (поля `clinic_*` из ТЗ 2.2): `id`(uuid),
`name`/`city`/`address`/`phone`/`working_hours`/`website`, `source`(idx), `lat`/`lng`
(карта), `rating` (фильтр), `has_online_booking` (фильтр), `created_at`. **Unique
`(name, city, source)`** — дедуп клиник по источнику.

**`RawPrice`** — сырой слой (ТЗ 3.1): `source`, `source_url`, `clinic_name_raw`, `city`,
`service_name_raw`, `price_raw` (строка как есть), `currency`, `duration_days`,
**`payload`(JSON, вся исходная строка для аудита)**, `fetched_at`(idx — якорь ретенции),
`processed`.

**`Price`** — нормализованная дедуплицированная строка (слой UI). Точное соответствие ТЗ 2.2:

| Колонка | Тип | ТЗ 2.2 / роль |
|---|---|---|
| `clinic_id` | FK(clinics) | `clinic_id` |
| `service_id` | FK(services) **nullable** | `service_id` (null = не привязано) |
| `service_name_raw` / `service_name_norm` | String | как на сайте / нормализованное |
| `category` | String(32) idx | `category` |
| `price_kzt` | **Numeric(12,2)** | `price_kzt` (точные деньги, не float) |
| `currency` | String(8) | `currency` (KZT; USD сконвертирован) |
| `duration_days` | Integer | `duration_days` |
| `source` / `source_url` | String | провенанс |
| `parsed_at` | DateTime(tz) idx | `parsed_at` (якорь свежести, ТЗ 4) |
| `is_active` | Boolean idx | `is_active` (снимается при устаревании, ТЗ 4) |
| `dedup_key` | String(64) **unique** idx | стабильный ключ upsert между прогонами (ТЗ 3.1) |

Деньги — `Numeric(12,2)` (а не `Float`) ради точности. Два составных индекса под горячие
пути поиска: `(service_id, is_active)` и `(category, is_active)`.

**`PriceHistory`** (ТЗ 3.4) — append-only наблюдения: `clinic_id`, `service_id`,
`price_kzt`, `recorded_at`(idx). Питает график истории.

**`UnmatchedQueue`** (ТЗ 3.2) — очередь ручной разметки: `service_name_raw`, `source`,
`occurrences`, `suggested_service_id`, `suggested_score`, `status`(pending/resolved).
**Unique `(service_name_raw, source)`** — одинаковые непривязанные имена не дублируются, а
наращивают `occurrences`. Сюда нормализатор кладёт всё, что ниже порога 86, **с подсказкой**.

**`ParseLog`** (ТЗ 3.1) — журнал прогонов: `source`, `status`(success/partial/error),
`message`(причина), `records_found/new/updated`, `errors_count`, `started_at`/`finished_at`.

**`Doctor`** (клон idoctor) — целочисленный PK = id источника; поля рейтинга/отзывов/опыта,
`spec_aliases`(Text, `",alias,"` для LIKE-фильтрации), JSON-списки специальностей/клиник/
болезней/услуг/отзывов, ленивые профильные поля + `profile_fetched`.

**`Illness`** (каталог «Лечение заболеваний») — `alias`(unique), `name`, JSON `skills`
(лечащие специальности) и `similar`, `spec_aliases`.

**`Subscription`** (ТЗ 3.4) — `email`, `service_id`, `clinic_id?`, `target_price_kzt?`,
`active`, `created_at`.

### 2.5 Pydantic-схемы (`schemas.py`)

`ParseRequest{sources?, include_live}`, `SubscriptionCreate{email, service_id, clinic_id?,
target_price_kzt?}`, `ResolveUnmatched{service_id, add_synonym=true}`,
`ChatTurn/ChatRequest{message, locale, history}`, `ServiceOut{...}`. Pydantic v2 даёт
автоматическую валидацию тел запросов и схему OpenAPI без ручного кода.

### 2.6 Сидинг и bootstrap (`seeding.py`)

- `load_dictionary(db)` — идемпотентный upsert справочника из `services_dictionary.json`
  по уникальному `code`.
- `seed_database(db, include_live=False)` — `load_dictionary` → `run_sources(...)` →
  `backfill_history` → `mark_stale_inactive` (поддержание правила 30 дней). Источники для
  наполнения выбираются вызывающей стороной; активный набор данных в этом проекте строится
  из **собственного справочника** (`seed`) + образцовых документов (`fixtures`); веб-скрейперы
  подключаются по `include_live=True`.
- `bootstrap_if_empty(db)` — first-boot guard из `lifespan`: по отдельности досоздаёт
  справочник, цены, врачей (`load_doctors`), болезни (`load_illnesses`), если их таблицы
  пусты. Живые скрейперы при автозапуске **не** дёргаются (только по явному флагу).

### 2.7 CLI (`cli.py`)

`python -m app.cli <init|seed|parse|stats>` (argparse). `seed [--live]`, `parse
--source ... [--live]` (выбор источников из реестра), `stats` (счётчики
клиник/услуг/активных цен/привязанных/в очереди). Это «ручной запуск парсинга через
интерфейс» из ТЗ 3.1 (наряду с админ-эндпоинтом).

## 3. Парсер / сбор данных (ТЗ 3.1)

Вся подсистема ингеста — под `backend/app/parsers/`. Построена вокруг **единого контракта
`RawRecord`**, который эмитит любой источник, и **единого пайплайна** (`pipeline.py`),
владеющего нормализацией, дедупом, слоями raw/normalized, журналом ошибок и ретенцией.
Источники — плагины через реестр; ядро не меняется при добавлении источника (ТЗ 4).
Сквозные форматы: **HTML, PDF, DOCX, XLSX/XLSM, XLS** + детерминированный локальный
генератор из собственного справочника.

### 3.1 Единый контракт (`base.py`)

- **`RawRecord`** (dataclass) — то, что эмитит любой парсер независимо от формата. Это шов,
  разделяющий *получение* (зависит от формата) и *обработку* (формат-независима). Поля:
  `source`, `service_name_raw`, `price_raw` (оба — **строки как спарсено**, парсинг позже —
  это и есть сырой слой ТЗ 3.1), метаданные клиники (name/city/address/phone/hours/website/
  lat/lng/rating/online), `currency`(KZT), `duration_days`, `source_url`, свободный
  `payload`(dict) для провенанса.
- **`parse_price()`** — единый разборщик «грязной» цены в `float|None`. Берёт первый
  числовой прогон (`\d[\d\s.,]*\d`), снимает все пробелы (включая NBSP — это всегда
  разделитель тысяч), затем разруливает `.`/`,`: при обоих — десятичным считается правый
  разделитель; при одной запятой — десятичной, если хвост 1–2 цифры, иначе тысячи;
  аналогично для точки. Примеры: `"от 3 500 тг"→3500`, `"1.200 ₸"→1200`, `"1 200,50"→1200.5`,
  `"99.99"→99.99`. Конвертацию валют делает не он, а `_to_kzt` в пайплайне.
- **`BaseParser`** — база сетевых парсеров: ленивый `httpx.Client` с самоидентифицирующимся
  `User-Agent`, `request_delay = 1.2 с` (вежливость, ТЗ 8), разбор `robots.txt`
  (`RobotFileParser`, **первый GET тоже ждёт задержку**), honoring `Crawl-delay`, и
  «вежливый GET», который при запрете robots или сетевой ошибке возвращает `None`, **никогда
  не бросая исключение**. Несетевые парсеры (`seed/real/fixtures`) наследуют контракт, но
  `client=None`.

### 3.2 Реестр-плагин (`registry.py`)

```python
PARSERS = {"seed": SeedParser, "fixtures": FixtureParser, "real": RealPricesParser,
           "kdl": KdlParser, "invitro": InvitroParser, "helix": HelixParser}
LIVE_SOURCES = ["kdl", "invitro", "helix"]
```

`get_parser(source)` инстанцирует класс по ключу. **Добавление источника = одна строка** в
`PARSERS` (и в `LIVE_SOURCES`, если сетевой). CLI, админ-роутер и планировщик читают эти две
структуры — ядро не трогается (ТЗ 4 расширяемость). Это прямой ответ на ТЗ 4 «архитектура
должна допускать добавление новых источников без переработки ядра».

### 3.3 Сквозной пайплайн (`pipeline.py`)

`run_source(db, source)` для одного источника:

1. **ParseLog открывается** заранее — журнал/ошибки (ТЗ 3.1).
2. **Сбор с изоляцией сбоев**: `get_parser(source).collect()` в try/except. **Любое**
   исключение → `status="error"`, причина `"{ExcType}: {msg}"`, `errors_count=1`, commit,
   return — батч не падает (ТЗ 4). `run_sources` гоняет источники последовательно, и
   падение одного не останавливает остальные.
3. Строится **нормализатор из живой БД** + кэш клиник.
4. **Цикл по записям** (каждая в своём try/except, одна плохая строка лишь увеличивает
   `errors`):
   - **Сначала сырой слой**: пишется `RawPrice` с нетронутым именем/ценой/payload (ТЗ 3.1).
   - **Парс+валидация цены**: `parse_price`; `None`/`≤0` → `errors`+continue; `_to_kzt`
     (USD×470).
   - **Upsert клиники** (`_ClinicCache`, кэш по `(name, city, source)`).
   - **Нормализация**: `normalizer.match(...)`; при отсутствии совпадения — `_queue_unmatched`
     (наращивает `occurrences` или создаёт запись очереди с подсказкой) (ТЗ 3.2).
   - **Ключ дедупа**: `sha1("{clinic_id}|{norm}|{source}")`, где идентичность —
     `match.service_id or raw_name`: при совпадении используется канонический `service_id`,
     поэтому одна услуга под двумя синонимами в одной клинике схлопывается в **одну** строку
     `Price` (ТЗ 3.1).
   - **Upsert в `Price` + `PriceHistory`**: ищем `Price` по `dedup_key`. Новая → вставка +
     запись истории. Существующая → обновление метаданных и `parsed_at`/`is_active`; запись
     истории **только если цена реально изменилась** (питает график, ТЗ 3.4).
5. **Финализация лога**: `records_found/new/updated`, `errors_count`, статус
   success/partial/error, `finished_at`, один `db.commit()`.

**Ретенция и свежесть:** `purge_old_raw` удаляет `RawPrice` старше `raw_retention_days`(90д)
(ТЗ 4); `mark_stale_inactive` снимает `is_active` с `Price` старше `stale_days`(30д) —
правило «не показывать данные старше 30 дней как актуальные» (ТЗ 4); поиск фильтрует по
`is_active`.

**Запуск (ручной + cron, ТЗ 3.1):** CLI `parse`/`seed`; админ `POST /api/admin/parse`;
планировщик `scheduler.py`. `--live` всюду означает «добавить `LIVE_SOURCES`».

### 3.4 Извлечение из реальных файлов (`real_extract.py` + `real_prices.py`)

Адаптер для прайс-листов клиник в форматах `.xlsx .xlsm .xls .docx .pdf`:

- **Per-format загрузчики**: xlsx→`openpyxl`, xls→`xlrd`, docx→`python-docx` (таблицы,
  fallback на абзацы), pdf→`pdfplumber` (`extract_tables`, fallback на построчный текст).
- **Колоночное распознавание** (`_find_columns`): сканирует первые 30 строк, находит колонку
  «Наименование» и ценовую (`цена/стоимость/тенге`, не «без»), **предпочитая колонку для
  граждан РК/резидентов** — ровно то, что нужно агрегатору цен для граждан РК.
- **Построчное извлечение** для PDF-текста/DOCX-абзацев: снимает лидирующий тариф-код,
  берёт имя до первой адекватной цены.
- **Валидация** `_valid_name` (5–110 символов, ≥3 кириллических, старт с заглавной/цифры,
  без слов-заголовков) и `_prices_in` (числа в `[300, 1_500_000]`).
- `RealPricesParser` читает сайдкар `_clinics.json` (файл→метаданные клиники), кап
  `PER_CLINIC_CAP=400` строк на клинику, изоляция на уровне файла (битый файл пропускается).

### 3.5 Детерминированный генератор из своего справочника (`seed_source.py`)

`SeedParser` генерирует реалистичные записи из **собственного** `services_dictionary.json` +
`clinics.json` (явно **не** из внешних сайтов). Назначение — прогнать **весь** пайплайн
нормализации детерминированно:

- **Детерминизм**: любой «случайный» выбор — стабильный `md5(parts)`→[0,1). Повторный прогон
  даёт байт-идентичные данные, что и стабилизирует демо, и демонстрирует дедуп (повторы
  схлопываются, ТЗ 3.1).
- **CAPS по профилю** клиники (lab/diagnostic/multiprofile) ограничивают число услуг на
  категорию — реалистичная выборка из 1200+ справочника.
- **Прогон нормализации через синонимы**: с вероятностью 65% эмитится **синоним** услуги, а
  не каноническое имя — нормализатор обязан сопоставить синоним обратно к id, как на реальном
  скрейпе.
- **Питание очереди**: с вероятностью 50% на клинику вбрасывается заведомо-неизвестное имя
  (напр. «Чек-ап Здоровое сердце») → попадает в `UnmatchedQueue` (ТЗ 3.2).

### 3.6 Образцовые документы и веб-скрейперы

- **`fixtures.py` + `file_extract.py`** — `FixtureParser` читает `data/fixtures/_clinics.json`
  и гоняет образцовые публичные прайсы (PDF/XLSX/DOCX) через `extract_file` (более простой
  экстрактор: pdfplumber/openpyxl/python-docx, ленивый импорт тяжёлых либ). Это сквозная
  демонстрация мультиформатного ингеста (ТЗ 3.1).
- **`html_extract.py` + `live/{kdl,invitro,helix}.py`** — веб-скрейперы. Извлечение
  **эвристическое** (а не CSS-селекторы под сайт): `PRICE_RE` (число + `₸|тг|тенге|kzt`),
  стратегия 1 — таблицы (ячейка-цена + ячейка-имя), стратегия 2 — элементы с ценой +
  ближайшее имя в предыдущих текстовых узлах/заголовках. Каждый сайт — список
  `catalog_paths`, пробуемых по очереди; всё идёт через вежливый `BaseParser.get` (robots +
  crawl-delay, ТЗ 8). Запускаются только по `--live`; JS/гео-ограниченные сайты просто дают
  меньше записей, прогон логируется `partial` и продолжается (ТЗ 4).

## 4. Нормализация и единый справочник (ТЗ 3.2)

Превращает разнородные сырые имена в единый справочник. Живёт в `normalization/`.

### 4.1 Нормализация текста (`text.py`)

`normalize_text(value)` применяется **идентично** к ключам справочника при построении
индекса и к каждому входящему имени/запросу — сравнение «на равных» и делает обычный
`dict`-lookup осмысленным точным матчем. Шаги: trim+lower → `ё→е` → NBSP→пробел → снятие
пунктуации (`[^\w\s]`, Unicode — кириллица выживает) → удаление шумовых токенов (`стоимость,
цена, услуга, анализ крови на, исследование, …, руб, тенге, тг`) → схлопывание пробелов.
Так `"Клинический анализ крови (ОАК), стоимость"` и `"ОАК — клинический анализ крови, цена,
тенге"` сводятся к близким строкам.

### 4.2 Матчер (`normalizer.py`) — дёшево → дорого

- **Индекс**: `{normalize_text(key) → _Entry}` по `[name, *synonyms]`, first-wins; кэш ключей.
- **`match(raw_name)`**: (1) пустой → none; (2) **точный** хит в индексе → score **100**,
  `exact` (O(1)); (3) **нечёткий** — `rapidfuzz process.extractOne(scorer=fuzz.WRatio)`.
  `WRatio` — взвешенная композиция (token-sort/token-set/partial), устойчива к перестановке
  слов и разнице длин — идеально для «общий анализ крови» vs «анализ крови общий». Если
  `score ≥ 86` → match; если ниже — **возвращает лучшего кандидата, но с `service_id=None`** и
  `matched=False`, чтобы наполнить очередь разметки **конкретной подсказкой**, а не пустотой.
- **`build_normalizer_from_db(db)`** — индекс строится из **живой** таблицы `services` при
  каждом вызове, поэтому правка синонима/имени через админку **немедленно** влияет на
  следующий матч — без кэша и редеплоя. Цена — пересборка индекса на каждый `resolve`
  (приемлемо для маленького справочника).

### 4.3 Замкнутый цикл ручной разметки (ТЗ 3.2)

Сырое имя нормализуется один раз и матчится к `service_id` на этапе парсинга; матчи ≥86
авто-привязываются, ниже-пороговые несут подсказку в `UnmatchedQueue`. Оператор в админке
жмёт «Привязать» → `POST /api/admin/unmatched/{id}/resolve`, который **учит синоним**
(добавляет сырое имя в `service.synonyms`) **и переподвязывает** все существующие `Price` с
этим сырым именем (ставит `service_id/service_name_norm/category`). Это полный цикл
«разметить → выучить → backfill».

## 5. Поиск, фильтры, сортировка, сравнение (ТЗ 3.3 / 3.4)

`search.py` + роутеры `search/services/meta`.

### 5.1 Автодополнение (ТЗ 3.3)

`autocomplete(db, q, limit)` — лёгкий матчер (без rapidfuzz): скорит услуги по
нормализованным имени+синонимам (точно=100, префикс=80, подстрока=60), **сортирует по
(score, число активных предложений)** — среди равных всплывает услуга с бОльшим числом
предложений. Возвращает `id/code/name/category/specialty/offers_count`.

### 5.2 Поиск предложений `search_offers` (ТЗ 3.3/3.4)

1. **Резолв услуги**: `service_id` напрямую, иначе `resolve_service(q)` (нормализатор →
   fallback на топ автодополнения).
2. **Базовый запрос**: `Price ⨝ Clinic`, фильтр `is_active=True` (неактивные/устаревшие не
   показываются никогда).
3. **Скоупинг**: по `service_id` (точно), иначе `ILIKE` по `service_name_norm`/`raw` (ловит
   непривязанные цены).
4. **Фильтры (ТЗ 3.3):** город, категория, `price_min/max`, `rating_min`, `online_booking` —
   каждый применяется только если задан.
5. **Свежесть (ТЗ 4):** `stale_cutoff = now - 30д`; каждое предложение получает `is_stale`, и
   устаревшие **отбрасываются по умолчанию** (если не `include_stale`). Считается
   `distance_km` (haversine, если есть геолокация).
6. **Сортировка (ТЗ 3.3):** `price_asc`(деф.)/`price_desc`/`updated`/`rating`/`distance`
   (только при заданном `lat`; точки без координат уходят в конец).
7. **Статистика сравнения (ТЗ 3.4):** `count/min/max/avg` по отфильтрованному не-устаревшему
   множеству.
8. **Пагинация** + ответ `{service, query, stats, offers}`.

Дизайн-решение: фильтрация по свежести, расстоянию, сортировка и статистика делаются
**в Python** над материализованным множеством — это держит haversine/staleness/stats простыми
и **БД-агностичными** (одинаково на SQLite и PostgreSQL). В SQL уходят только `is_active`,
услуга/текст и колоночные фильтры.

### 5.3 Эндпоинты справочника и мета

- `GET /api/search/autocomplete`, `GET /api/search` (полный набор параметров → `SearchParams`).
- `GET /api/services` (фильтр категория/`q`, сорт `name`/`popular`), `GET /api/services/{id}`
  (карточка услуги + `min/max/avg` цен — независимый от клиник разброс).
- `GET /api/meta` — **всё для фильтр-бара и дашборда одним запросом**: города, источники,
  категории+метки, счётчики, `price_range`, **`last_updated`** (`max(parsed_at)` — сигнал
  свежести, ТЗ 3.3/4), `server_time`.

## 6. Прочие API-функции

Все роутеры тонкие: валидируют вход, запрашивают модели, формируют JSON. Общий `_iso(dt)`
форсит UTC перед `.isoformat()` (консистентные таймстампы свежести, ТЗ 4).

### 6.1 Карточка клиники (`clinics.py`, ТЗ 3.3)
- `GET /api/clinics` — список с фильтрами город/источник/онлайн; число услуг на клинику —
  **одним сгруппированным запросом** (без N+1). Двойного назначения: данные для карты и
  фильтров.
- `GET /api/clinics/{id}` — **карточка клиники**: все активные услуги, упорядоченные по
  `(категория, имя)` (фронт группирует по категориям), контакты, `parsed_at` (последнее
  обновление), флаг `matched` (привязано ли к справочнику, ТЗ 3.2).

### 6.2 История и сравнение (`history.py`, ТЗ 3.4)
- `GET /api/history` — временной ряд из `PriceHistory`, **сгруппированный по клиникам** в
  серии (готово для мульти-линейного графика).
- `GET /api/compare` — **таблица сравнения**: делегирует `search_offers` (единый движок
  ранжирования → консистентность с поиском), опционально фильтрует `clinic_ids`, ставит флаг
  `is_cheapest` и **пересчитывает stats** по отфильтрованному набору.

### 6.3 Админка / управление парсером (`admin.py`, ТЗ 3.1/3.2)
- `POST /api/admin/parse` — **ручной запуск** (source-список, `include_live`, фон/синхронно);
  фон не блокирует запрос (ТЗ 4).
- `GET /api/admin/parse-logs` — журнал прогонов (ТЗ 3.1).
- `GET /api/admin/unmatched` — очередь, отсортированная по `occurrences` (частые имена
  наверх).
- `POST /api/admin/unmatched/{id}/resolve` — замкнутый цикл (выучить синоним + переподвязать
  цены).
- `GET /api/admin/stats` — дашборд (клиники/услуги/активные/raw/привязанные/в очереди +
  разрезы по городу/источнику/категории; `raw_rows` vs `active_prices` делает двухслойность
  наблюдаемой).

### 6.4 Подписки (`subscriptions.py`, ТЗ 3.4)
`POST` создать, `GET ?email=` список активных, `DELETE` — мягкое удаление (`active=False`).

### 6.5 Карта (`map.py`, ТЗ 3.4)
`GET /api/map/cities` — на город минимальная цена + счётчики (одним сгруппированным запросом),
координаты из `REGION_COORDS` (18 городов) — «бабблы» в стиле Aviasales (дешевейшее «от X»).

### 6.6 Каталог врачей (`doctors.py`) — клон idoctor.kz
`meta` (регионы/специальности), `GET /api/doctors` (фильтры регион/специальность/цена/рейтинг/
опыт/онлайн + пагинация; специальность через `spec_aliases LIKE '%,alias,%'` — точный матч без
join-таблицы; сорт `rating` = композит `top→verified→rating`), `recommendations`
(умные «выгодные предложения»: bestValue/cheapest/topRated/popular с % ниже среднего),
`{id}` и `{id}/profile` (**ленивая** дозагрузка профиля+отзывов с idoctor + кэш). Загрузка 40k
врачей — `bulk_insert_mappings` батчами по 4000.

### 6.7 Каталог болезней (`illnesses.py`)
`categories` (инверсия болезнь→специальности), `GET /api/illnesses` (поиск/категория), `{alias}`
(болезнь + лечащие специальности + **врачи, которые её лечат** через
`or_(spec_aliases LIKE '%,a,%' ...)`, опц. фильтр региона). 274 болезни.

### 6.8 Ассистент (`assistant.py` + `llm.py`)
**Детерминированный движок интентов/сущностей поверх живой БД** + опциональный рерайт через
OpenAI с полным фолбэком: сначала всегда считается детерминированный ответ (`det`); без ключа
OpenAI — отдаётся он; с ключом — строится **строгий системный промпт** с `FACTS: {det.reply}`
и запретом «выдумывать данные/вставлять URL» — **LLM лишь переписывает прозу, ссылки-кнопки
всегда из заземлённых данных**. Движок различает интенты (специальность/клиника/«где дешевле»/
врач/цена услуги/город), резолвит услугу 3-уровнево (аббревиатура→containment→`token_set_ratio`
≥82) и **наследует контекст** из истории («а в Караганде?»). `llm.py` возвращает `None` на
любой не-200/исключение — ассистент никогда не падает из-за LLM.

### 6.9 Планировщик (`scheduler.py`, ТЗ 3.1 cron / ТЗ 4)
In-process `APScheduler` (UTC), gated `enable_scheduler`. `_daily_job`: `run_sources(["seed"])`
→ `mark_stale_inactive` (правило 30 дней) → `purge_old_raw` (ретенция 90 дней), весь блок в
try/except — упавший прогон не убивает поток (ТЗ 4). Пара к ручному `POST /api/admin/parse`
закрывает требование ТЗ 3.1 «ручной + по расписанию».

### 7.3 Ключевые страницы

  (degrade на пустые при сбое бэка), герой-градиент, большой `SearchBar`, 4 живых счётчика,
  карточки категорий, сетка популярных услуг с `last_updated`.
- **Поиск** (`search/page.tsx`, ядро ТЗ 3.3): `SearchBar` с **автодополнением** (debounce 180
  мс, клавиатурная навигация), `Filters` (сорт/город/категория/цена/рейтинг/онлайн/«показывать
  устаревшие»/«рядом со мной» геолокация → авто-сорт по расстоянию), debounce 250 мс с отменой
  устаревших ответов, список `OfferCard` (клиника→карточка, рейтинг/онлайн-чипы, адрес/часы/
  расстояние, **бейдж свежести** красный при `is_stale`, цена, кнопка подписки, «На сайт»,
  «В сравнение»), липкая панель сравнения, `SubscribeModal` (ТЗ 3.4).
- **Сравнение** (`compare/page.tsx`, ТЗ 3.4): пикер услуг или таблица. **Аккордеон сетей
  клиник** — отличительная фича: `chainGroups` группирует предложения по `source||clinic_name`
  (Map сохраняет порядок → наследует глобальную сортировку); `splitChain` режет имя по тире
  («Медцентр Olymp — Гагарина» → сеть «Медцентр Olymp», филиал «Гагарина»). Одиночная клиника
  — обычная строка; сеть из нескольких филиалов — **раскрываемая родительская строка** с
  агрегатом («от {min} ₸», N филиалов, число городов, лучший рейтинг, есть-ли-онлайн); самая
  дешёвая сеть **раскрывается по умолчанию**; «дешевле всех» подсвечивает строку. Ниже —
  `PriceHistoryChart` (Recharts, перенос последней известной цены вперёд для ступенчатого
  графика).
- **Карточка клиники** (`clinics/[id]`): серверная; услуги по категориям, мини-карта
  (`ClinicMiniMap`, MapLibre, `ssr:false`), **маршрут 2GIS** (ТЗ 3.4), метка «не привязано к
  справочнику» при `!matched`.
- **Карта** (`map/page.tsx`+`CityPriceMap`): **MapLibre GL**, бесплатный векторный стиль CARTO
  (без ключа); HTML-маркеры — «бабблы» цен по городам (дешевейший — teal) + точки клиник;
  `flyTo`/`fitBounds`; `CitySelect` (комбобокс без зависимостей).
- **Врачи / болезни**: клон idoctor (фильтры/пагинация/рекомендации/профиль с отзывами,
  санитайз HTML описания, 2GIS-гео) и каталог «Лечение заболеваний» с селектором города.
- **Админка** (`admin/page.tsx`): дашборд (счётчики, **raw-слой**), ручной запуск парсинга,
  очередь разметки с «Привязать», журнал прогонов, диаграммы по источникам/категориям.

### 7.4 i18n и адаптивность
`I18nProvider` (RU/KK/EN): restore из `localStorage` или **автодетект** из `navigator.language`,
`t(key, vars)` с фолбэком на RU; `messages.ts` — три словаря (~150 ключей; **данные** —
имена врачей/клиник — остаются на языке источника, переводится только UI). **Адаптивность** —
на брейкпойнтах Tailwind: `lg:grid-cols-[280px_1fr]` схлопывается в одну колонку, навигация
прячет подписи ниже `md`, таблица сравнения в `overflow-x-auto` с липкой первой колонкой,
тяжёлые карты — `next/dynamic ssr:false` со скелетонами (ТЗ 3.3 «корректное отображение на
мобильных»).

## 8. Данные и оффлайн-скрипты: как строится справочник

Оффлайн «фабрика данных» (`scripts/*.py`) превращает источники в статические JSON под `data/`,
которые `seed_database()` грузит в БД. Ничего из этого не работает в рантайме.

### 8.1 Собственный справочник услуг (`build_dictionary.py` → `services_dictionary.json`)
Это сердце нормализации (ТЗ 3.2), **построенный из официального справочника ТЗ**
`data/source/services_reference.xlsx` (колонки `ID, Специальность, Code, Name_ru,
TarificatrCode`, 1287 строк). Алгоритм: дедуп по нормализованному имени (1286→**1230
услуг**); **категория из префикса тариф-кода** (`A`=врач, `B/В`=лаборатория, `C`=диагностика,
`D`=процедура; иначе по ключевым словам имени) — распределение `laboratory 721 / procedure 230
/ diagnostic 180 / doctor 99`, **122 специальности**; авто-синонимы (скобочные аббревиатуры,
варианты «Приём»/«Консультация»); эвристическая базовая цена (детерминированный jitter);
**курируемый overlay** `curated_overlay.json` (69 записей, напр. ОАК с 8 синонимами вкл. CBC),
матч точный-или-rapidfuzz ≥90, помечает **40 core-услуг**. Запись содержит
`code/official_id/name/specialty/tarif_code/category/category_label/synonyms/base_price_kzt/
duration_days/is_core`. 1230 услуг / 122 специальности **существенно превышают** требование
ТЗ «≥50 нормализованных позиций».

### 8.2 Реестр клиник (`build_clinics.py` → `clinics.json`)
Курируемый реестр **19 филиалов** по **6 городам**, **8 брендов** (kdl/invitro/helix/olymp/
medel/mck/aksai). Знобы реализма: `profile→categories` (что клиника «держит») и `price_factor`
0.85–1.15 (премиум↔бюджет — основа сравнения); `jitter()` детерминированно разносит координаты
филиалов, чтобы маркеры не накладывались (ТЗ 3.4).

### 8.3 Мультиформатные фикстуры (`build_fixtures.py` → `data/fixtures/`)
Генерирует **3 реалистичных публичных прайса в 3 форматах** (PDF/XLSX/DOCX), чтобы сквозно
проверить мультиформатный ингест (ТЗ 3.1); намеренно выбирает **синоним** имени, чтобы
нормализатору было что делать; PDF пишет кириллическим TTF (иначе текст pdfplumber — мусор).

### 8.4 Внешние датасеты — врачи и болезни
Скрейпятся с **idoctor.kz**: `data/doctors/doctors.json` — **40 467 врачей** (+ `specialties.json`
237, `regions.json` 18) через публичный JSON-API с пагинацией, **бэкофф + обновление
Cloudflare-cookie**, атомарная докачка (tmp+`os.replace` — крах не рвёт датасет, ТЗ 4);
`data/diseases/diseases.json` — **274 болезни** (BFS по графу `similar` от разнообразных seed).

### 8.5 Реальные прайс-файлы (опциональный адаптер)
`data/real_prices/` содержит реальные файлы клиник + `_clinics.json`, а `RealPricesParser` —
подключаемый адаптер для их прямого ингеста (см. 3.4); `inspect_real_prices.py` — read-only
диагностика раскладки файлов. В активном наборе данных проект строится из **собственного
справочника** (`seed`) + образцовых документов (`fixtures`), а прямой ингест клиентских файлов
и веб-скрейперы — расширяемые опции реестра.

## 9. Docker и развёртывание

Один `docker compose up --build` поднимает три сервиса (ТЗ 5):

- **db** — `postgres:16-alpine`, том `pgdata`, **healthcheck** `pg_isready` (бэкенд ждёт
  готовности БД через `depends_on: condition: service_healthy`).
- **backend** — `backend/Dockerfile` (`python:3.12-slim`): ставит `requirements.txt`, копирует
  `backend/app` + `data`, `uvicorn app.main:app`. Env: `DATABASE_URL=postgresql+psycopg2://…`,
  `DATA_DIR=/app/data`, `ENABLE_SCHEDULER=false`. На первом старте `bootstrap_if_empty` сам
  создаёт таблицы и наполняет БД.
- **frontend** — `frontend/Dockerfile`, **многостадийная сборка** (deps → build → run) на
  `node:20-alpine` + `pnpm` (corepack): билдит Next и запускает `pnpm start`. `NEXT_PUBLIC_API_URL`
  прокидывается и как build-arg (для клиента), и как env.

Порты: 3000 (фронт), 8000 (бэкенд + Swagger `/docs`), 5432 (Postgres). Локально без докера —
SQLite «из коробки» (`python -m app.cli seed` + `uvicorn`, `pnpm dev`).

**Почему так:** кросс-СУБД движок + env-конфиг означают, что **тот же код** идёт и в
zero-infra dev (SQLite), и в прод (Postgres) — без ветвлений в коде. Многостадийный фронт-образ
не тащит dev-зависимости в рантайм. Healthcheck + `depends_on` убирают гонку старта.

## 10. Нефункциональные требования (ТЗ 4)

| Требование ТЗ 4 | Как выполнено |
|---|---|
| **Актуальность ≥1/сутки** | `APScheduler` `_daily_job` (cron 03:00) + ручной `POST /api/admin/parse`; идемпотентный upsert по `dedup_key` — повторный прогон не плодит дубли. |
| **Ответ UI ≤3 с** | Поиск — индексы `(service_id,is_active)`/`(category,is_active)`, агрегаты одним сгруппированным запросом (без N+1), маленький справочник в памяти, debounce на фронте, `cache:no-store`. На демо-наборе ответы — десятки мс. |
| **Не показывать >30 дней как актуальные** | `parsed_at` + `is_active`; `mark_stale_inactive` снимает `is_active`; поиск фильтрует `is_active` и помечает `is_stale`; UI рисует красный бейдж и скрывает устаревшее по умолчанию. |
| **Добавление источников без переработки ядра** | Реестр-плагин `PARSERS` + единый `RawRecord` → новый источник = один класс + одна строка. |
| **Отказоустойчивость парсера** | Изоляция на уровне источника **и** записи (try/except), `ParseLog(status=error, message)`; падение одного источника/файла/строки не валит остальное; вежливый GET возвращает `None`, не бросая. |
| **Хранение сырых ≥90 дней** | Отдельный слой `raw_prices` (`payload` JSON, `fetched_at`); `purge_old_raw` удаляет старше `raw_retention_days=90`. |

## 12. Соответствие ТЗ и почему решение оптимально

### 12.1 Карта требований → реализация

| Раздел ТЗ | Реализация | Почему так — технически лучшее |
|---|---|---|
| **2.2 Схема** | Модели `Clinic`/`Price`/`Service` 1:1 покрывают поля; деньги `Numeric(12,2)`; `parsed_at`/`is_active` | Точный денежный тип (не float — без ошибок округления); схема прямо отражает ТЗ, плюс служебные таблицы под 3.1/3.4 |
| **3.1 Парсер** | `RawRecord`-контракт + `pipeline` + реестр; форматы HTML/PDF/DOCX/XLSX/XLS; raw-слой; `ParseLog`; ручной+cron | Шов «получение↔обработка» делает добавление формата/источника локальным; raw отдельно → аудит и идемпотентность |
| **3.2 Нормализация** | Свой справочник (1230 из офиц. реестра) + 2-уровневый матчер (exact→WRatio≥86) + очередь с подсказками + замкнутый цикл resolve | rapidfuzz WRatio устойчив к перестановкам слов; ниже-пороговое не теряется, а идёт человеку с подсказкой; индекс из живой БД → правки сразу влияют |
| **3.3 Поиск/UI** | Автодополнение, 5 фильтров, 5 сортировок (вкл. haversine-расстояние), карточка клиники, дата обновления, адаптив | Сервер-сайд фильтры + клиентский debounce; «последнее обновление» прозрачно; адаптив на Tailwind |
| **3.4 Доп.** | Карта (MapLibre, бабблы), подписки, сравнение-таблица (+ аккордеон сетей), история (Recharts), маршрут 2GIS | MapLibre — авиасейлс-вид без ключа/затрат; аккордеон сетей решает реальную проблему «разбросанных филиалов» |
| **4 Нефункц.** | См. §10 | Индексы, изоляция сбоев, ретенция, staleness — всё заложено в модель данных, а не прикручено сбоку |
| **5 Стек** | Python+FastAPI+SQLAlchemy, Next.js+pnpm, PostgreSQL(docker)/SQLite(dev), APScheduler, Docker | Точно рекомендованный ТЗ стек; кросс-СУБД даёт и zero-infra, и прод |
| **6 Результаты** | README + рабочий MVP; справочник 1230 (≥50); ≥3 источника; тесты | Существенное превышение минимумов; машинно проверено тестами |
| **8 Правила** | robots.txt, `request_delay`, только открытые данные, без перс. данных | Вежливый `BaseParser` соблюдает robots + crawl-delay |

### 12.2 Почему ключевые архитектурные решения — наилучшие

1. **Двухслойное хранение (raw → normalized).** Большинство MVP пишут сразу в одну таблицу.
   Здесь сырьё неизменяемо хранится в `raw_prices` (с `payload` для полного аудита), а UI
   читает только чистый `prices`. Это даёт: аудит (ТЗ 4), идемпотентные повторные прогоны
   (upsert по `dedup_key`), независимые сроки жизни слоёв, и возможность перематчить старые
   данные без повторного скрейпа.
2. **Единый контракт `RawRecord` + реестр-плагин.** Прямой ответ на ТЗ 4 «без переработки
   ядра»: PDF, XLS, HTML, генератор — все сходятся в одну точку `pipeline.run_source`, поэтому
   ретенция, нормализация, дедуп, история, свежесть и журнал ошибок **единообразны** для
   любого источника. Новый источник — один класс.
3. **Нормализатор, пересобираемый из живой БД.** Правка синонима в админке мгновенно меняет
   будущие матчи — без кэша, без редеплоя. Замкнутый цикл `resolve` ещё и переподвязывает уже
   спарсенные цены — данные «самоисцеляются».
4. **Очередь разметки с подсказками вместо тихой потери.** Реальные каталоги содержат услуги,
   которых нет в тарификаторе. Вместо ложного матча (порог 86 специально консервативен) такие
   имена идут человеку **с конкретной fuzzy-подсказкой и счётчиком частоты** — приоритизация
   усилий разметки (ТЗ 3.2).
5. **Кросс-СУБД на SQLAlchemy 2.0.** Один код: SQLite для мгновенного локального старта,
   PostgreSQL в проде — переключение одной `DATABASE_URL`. Ни ветвлений, ни вендор-лока.
6. **Свежесть как свойство модели данных.** `parsed_at`+`is_active`+`stale_days` — правило
   30 дней встроено в запросы, а не «прикручено»: поиск физически не может вернуть устаревшее
   как актуальное.
7. **MapLibre без токена.** Авиасейлс-подобная векторная карта с ценовыми бабблами — **без**
   API-ключа и затрат (бесплатный стиль CARTO), в отличие от Google Maps/Mapbox.
8. **Детерминированный seed из своего справочника.** Демо-данные воспроизводимы (md5-seed),
   прогоняются через **тот же** пайплайн нормализации/дедупа, что и реальный скрейп, и
   намеренно подают синонимы и неизвестные имена — то есть честно нагружают 3.2, а не
   подкладывают готовые ответы.
9. **Ассистент, который не может соврать.** LLM переписывает только прозу; все числа и
   ссылки — из детерминированных, заземлённых в БД фактов; без ключа OpenAI всё работает на
   детерминированном движке. Надёжность важнее «магии».

## 13. Заключение

MedServicePrice.kz реализует **полный путь данных** — сбор (мультиформатный, расширяемый,
отказоустойчивый) → сырой слой → нормализация к собственному справочнику с человеко-в-цикле →
дедуплицированный слой выдачи → быстрый фильтруемый/сортируемый поиск со свежестью →
адаптивный UI с картой, сравнением (включая аккордеон сетей клиник), историей цен и
подписками. Каждое требование ТЗ (2.2, 3.1–3.4, 4, 5, 6, 8) имеет конкретную, наблюдаемую
реализацию, а ключевые решения (двухслойность, контракт-шов, живой нормализатор, кросс-СУБД,
свежесть-в-модели, MapLibre-без-ключа) выбраны так, чтобы быть одновременно **простыми,
расширяемыми и устойчивыми** — то есть технически оптимальными под букву и дух задания.

> Карты файлов и точные `file:line` — см. README.md и исходники; этот документ описывает
> **архитектуру и обоснование**, README — **запуск**.
